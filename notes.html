microbit-esp32-remote — project notes

ESP32 firmware that impersonates a BBC micro:bit so the unmodified rxy web app can drive it. Last updated 2026-05-15. Repo: abourdim/microbit-esp32-remote.

1. The two ESP32-C3 USB-CDC upload traps

The C3 Super Mini exposes its USB-C port through the chip's native USB peripheral — there is no CH340/CP2102 bridge. esptool's default flow makes two assumptions that this peripheral breaks.

#SymptomCauseFix
1 A fatal error occurred: No serial data received.
…right after Changing baud rate to 460800		 Mid-upload baud switch crashes the CDC endpoint. upload_speed = 115200 in platformio.ini (single rate end-to-end).
2 Unable to verify flash chip connection (No serial data received.)
…right after Stub running...		 When the flasher stub takes over from the ROM bootloader, the USB-CDC endpoint re-enumerates and the first post-stub command (read SPI flash JEDEC ID) lands in a dead window. upload_flags = --no-stub — keep the ROM bootloader in charge for the whole upload. Slower (~60–90 s) but reliable.
arduino-cli caveat. There is no clean way to pass --no-stub through an Arduino FQBN. If you must use arduino-cli on this board, hold the BOOT button while pressing RESET to force ROM-bootloader mode, then immediately upload. Recommended path: use PlatformIO.

2. BLE protocol — what rxy actually expects

Verified against abourdim/rxy@main:script.js on 2026-05-15.

2.1 UUIDs (Nordic-UART-style, with a twist)

RoleUUID
Service6e400001-b5a3-f393-e0a9-e50e24dcca9e
Notify (device → app) — micro:bit TX6e400002-…
Write (app → device) — micro:bit RX6e400003-…
The roles of 0002 and 0003 are the opposite of the Nordic UART Service convention used by Adafruit/Bluefruit. We mimic the micro:bit, which assigns them the other way around. The firmware declares the characteristics explicitly, so this is handled.

2.2 Wire protocol

DirectionMessageNotes
app → deviceGETCFGsent ~500 ms after connect
app → deviceSET <id> <val…>val may be multi-token (joystick, dpad, xypad)
device → appCFGBEGIN / CFG <18-char chunk>… / CFGENDchunk size 18 is mandatory — matches the MakeCode template
device → appUPD <id> <val>only after CFGEND has been sent

Encoding: ASCII, \n-terminated; \r tolerated. rxy rate-limits incoming messages to ~one per 200 ms — do not flood it with UPDs; 200–500 ms cadence is comfortable.

2.3 Advertising packet budget

BLE advertisement payload is 31 bytes. With a 128-bit service UUID (~18 bytes incl. headers) plus flags (3 bytes), the name does not fit in the primary packet. Solution: UUID in the primary, name in the scan response (adv->setName() with enableScanResponse(true)). rxy filters by namePrefix: 'BBC micro:bit', which matches names in either packet.

2.4 Name mismatch (open item)

README promises BBC micro:bit ESP32 as the advertised name, but the firmware advertises plain BBC micro:bit (main.cpp:52). Both pass the rxy namePrefix filter, but only the longer name lets you distinguish the ESP32 from a real micro:bit in the chooser. Not fixed yet.

3. Customising the layout

  1. Open rxy → Build tab → design the remote.
  2. Click 📄 Code. In the generated MakeCode, find const CFG = "…". Copy the long base64 string.
  3. Paste it into LAYOUT_CFG_BASE64 in the firmware, replacing the default minimal one-button layout.
  4. Re-upload.

Widget IDs (btn_fire, slider_speed, …) you chose in the Build tab arrive verbatim in SET messages; handle them by name in handleWidget().

4. Widget value formats

PrefixWidgetValue format
btn_Button0 | 1
slider_Sliderinteger in widget's min..max range

angle: 0°=right, 90°=down, 180°=left, 270°=up

Outputs (LEDs, labels, gauges, graphs, battery) are pushed with sendValue("<id>", "<val>") which emits UPD <id> <val>.

6. Open items from the code review (2026-05-15)

  1. cosmetic Device-name mismatch: README ↔ main.cpp:52. Pick one.
  2. robustness Add if (!gConnected) return; inside sendCfg()'s chunk loop (main.cpp:145-150) so a mid-CFG disconnect doesn't keep delaying.
  3. nice-to-have NimBLEDevice::setMTU(247); in setup() for longer UPD headroom (e.g. long label/graph strings).
  4. docs One-line BLE-security caveat in README: "no pairing — anyone in range can drive widgets".
  5. micro Convert gRxBuffer from String to char[256] to avoid heap churn over long sessions.

8. Caveats

  • Name spoofing. Firmware advertises a name with the BBC micro:bit prefix to satisfy the rxy filter. Any other micro:bit-aware tool may also try to connect.
  • No BLE pairing. The connection is open — anyone in radio range can write SET commands. Fine for a desk-toy; not for production.
  • One connection at a time. BLE peripherals hold a single central; reconnecting from another phone requires disconnecting first.
  • Native USB resets. A firmware crash takes the USB-CDC endpoint down with it; the host may need a re-plug.
